## Architecture

Crosshair is a convergence-executor daemon. It reads one or more `bullseye.yaml` target files, and for every target carrying a `strategy` block it runs the declared shell `command` on a tick loop, persisting per-target attempt state (last attempt, last success, consecutive failures, cooldown) in SQLite. It is designed to run under launchd `KeepAlive`.

Data flow:

```
bullseye.yaml (disk) → loader (canonicalize + parse) → StrategyTarget
    → runner (sh -c <command>) → AttemptOutcome
    → store (SQLite upsert) → status CLI (read-only join + table)
```

Key modules:

- **`src/main.rs` / `src/cli.rs`** — CLI (clap): `run` and `status` subcommands, repeatable `--config`, optional `--state` (default `$HOME/.local/state/crosshair/state.db`).
- **`src/tick.rs`** — `run()` opens the Store and drives either one-shot (`--once`) or a `tokio::time::interval` loop (`MissedTickBehavior::Delay`). `run_one_tick()` reloads all configs every tick and iterates targets **sequentially**, awaiting `converge_one` for each. `converge_one()` gates on `needs_attempt`, honours `cooldown_until`, runs the attempt, computes next cooldown via `BACKOFF_LADDER [30m, 2h, 6h, 24h]`, and records the outcome.
- **`src/runner.rs`** — `run_attempt()` spawns `sh -c <command>` with piped stdout/stderr, null stdin, `kill_on_drop(true)`; drains each pipe in a spawned task capped at 64 KiB; enforces `resolved_timeout` (per-strategy or `DEFAULT_TIMEOUT` 5m) via `tokio::time::timeout` around `child.wait()`.
- **`src/store.rs`** — rusqlite (bundled). Single table `target_state`, PK `(yaml_path, target_id)`. RFC3339 timestamps. `record_attempt()` upserts, re-reading the prior row to bump/reset `consecutive_failures`; stdout/stderr truncated to 4096 bytes on a char boundary.
- **`src/status.rs`** — opens the same DB read-only and prints a fixed-width status table.

Each tick is intended to be independent and idempotent: missed or stranded runs self-heal on the next tick.

## Invariants

The audit was framed against these system invariants (from recon). The findings below violate #1, #8, #10 (process/drain management) and #5 (backoff/state persistence).

1. **The per-attempt timeout must always terminate the child process.** This is the raison d'être of the project — the motivating bug was a 9.5h yadm-SSH hang under launchd. Any path that lets a strategy run unbounded is a critical regression.
2. Terminal-status targets (Achieved, SetAside) are never executed.
3. State rows are uniquely keyed by canonicalized `(yaml_path, target_id)`; run loop and status CLI must canonicalize identically.
4. `consecutive_failures` increments by exactly 1 per failure, resets to 0 on success; the two code paths deriving it must agree.
5. **Backoff is monotonic and capped; success clears `cooldown_until`.** This is what prevents both busy-loop hammering of a failing target and permanent starvation of a recovered one.
6. A strategy runs only when `needs_attempt()` AND (`cooldown_until` is None OR `<= now`).
7. Captured output is bounded (64 KiB read-time, 4096 B store-time, char-boundary truncation).
8. **Each tick is independent and idempotent; a slow tick delays but does not queue duplicate ticks, and correctness must not depend on any tick having fired on schedule.**
9. RFC3339 timestamps round-trip losslessly; an unparseable stored timestamp degrades to None and must not be read as "never attempted".
10. **`kill_on_drop(true)` must hold so a dropped attempt future leaves no orphaned child.**

## Trust boundaries

1. **`strategy.command` → arbitrary shell execution.** Dispatched through `sh -c` unquoted; anyone who can write a scanned config gains code execution as the daemon user. This is the primary authority-crossing surface and the common root of findings 1–3.
2. **YAML content → deserialization.** A parse/canonicalize error on any one config aborts the whole tick's load.
3. **Child stdout/stderr → SQLite.** Untrusted, potentially huge/binary output; bounds and char-boundary truncation are the defenses.
4. **Exit code / timeout status → success determination.** `Some(0)` = success; a command that exits 0 without doing the work is indistinguishable from success (intentional in v0.1).
5. **`$HOME` → state DB path.**
6. **CLI `--config`/`--state` + filesystem → canonicalization and state-dir creation.**
7. **humantime strings (`--tick`, `strategy.timeout`) → scheduling.** An unparseable per-strategy timeout silently falls back to the 5m default.
8. **Shared SQLite file between the `run` daemon and the `status` CLI.** Two OS processes, no explicit WAL or busy_timeout — the boundary at the heart of finding 4.

## Findings

### F1 — Timeout kills only the `sh` PID; post-kill drain `await` hangs the entire tick loop forever

- **Severity:** Critical
- **Verdict:** CONFIRMED (reachability: confirmed · invariant-validity: confirmed)
- **Location:** `src/runner.rs:124` (timeout arm); identical defect on the wait-error arm at `:110`. Root spawn at `:67-73`; unguarded drains at `:125-126` and `:111-112`; read loop at `:142-156`; sequential tick loop at `src/tick.rs:78-82`, awaited with no wrapper at `:98`.
- **Invariants violated:** #1 (timeout must terminate the child), #8 (a slow tick must not block subsequent ticks), #10 (no orphaned child). Trust boundary #1.

**Failure scenario.** The command is dispatched via `sh -c <command>` with no `.process_group(0)`. The canonical `yadm-auto-sync` strategy runs `sh → yadm → git → ssh`, where `ssh` inherits fds 1/2 (the pipe write-ends). On timeout, `child.kill().await` sends SIGKILL to the `sh` PID **only**; the re-parented `ssh` grandchild keeps running (the exact multi-hour hang the project exists to prevent — now orphaned and unbounded). Worse, because the surviving `ssh` still holds the pipe write-end open, the stdout/stderr pipes never reach EOF, so `read_to_string_capped` never returns `Ok(0)` and `stdout_task.await` / `stderr_task.await` block **indefinitely**. `run_attempt` therefore never returns, wedging the sequential tick loop on this one target forever — no other target is ever evaluated again and the daemon is effectively dead.

**Evidence.** `runner.rs:67-73` builds the `sh -c` Command with no `.process_group(0)`. `runner.rs:124` and `:110` call `child.kill().await` (single-PID SIGKILL, not a group kill). `runner.rs:125-126` and `:111-112` unconditionally `.await` the drain tasks. `runner.rs:142-156` returns only on cap/EOF/error. The per-attempt `timeout(limit, child.wait())` (`:96`) *does* fire because `sh` stays alive while its descendant blocks, so execution reliably reaches the defective kill+drain path. `tick.rs:78-82` runs targets in a sequential `for t in targets { converge_one(...).await }` loop; `tick.rs:98` awaits `run_attempt` with no outer timeout. Both lenses independently reached CONFIRMED; documented contract at `agents-guide.md:93` ("the per-attempt timeout bounds how long any single hung command can block subsequent ticks") and `README.md:9,28`. Adversarial refutations (sh exec-in-place, descendant redirecting fds, ssh self-timing-out, an outer drain timeout) all fail: SIGKILL never reaches grandchildren regardless of exec/fork, the defect holds for any pipe-inheriting descendant, and no timeout wraps the drain awaits.

**Fix sketch.** (a) Add `.process_group(0)` to the Command builder so `sh` and all descendants share a killable group. (b) On timeout and on the wait-error path, kill the whole group (`nix::sys::signal::killpg(Pid::from_raw(pgid), SIGKILL)`, or `kill -- -<pgid>`), not just `child.kill()`. (c) Do not unconditionally `.await` the drain tasks after a kill — wrap the joins in a short `timeout` and `abort()` on expiry, so a descendant holding the pipe open can never wedge the tick. Killing the whole group also closes all inherited write-ends, yielding prompt EOF for the drain.

**Verification step.** Add an integration test with a strategy whose command spawns a descendant that outlives its parent and keeps stdout open, e.g. `command: "sh -c 'sleep 3600 & exec sleep 0'"` (parent exits immediately; backgrounded `sleep` inherits the pipe) with a short `strategy.timeout` — and separately a command that itself hangs past the timeout while a grandchild holds the pipe. Assert that (1) `run_attempt` returns within `timeout + epsilon`, (2) `timed_out` is surfaced, (3) no descendant process survives (poll by pgid), and (4) the next target in the same tick is reached. Loop-wedge is proven by a second target's state row advancing in the same tick.

### F2 — Per-attempt timeout kills only the direct `sh` child; hung grandchildren survive as orphans

- **Severity:** High
- **Verdict:** CONFIRMED (reachability: confirmed · invariant-validity: confirmed)
- **Location:** `src/runner.rs:67-73` (spawn, no process group, `kill_on_drop` only), `:110` and `:124` (`child.kill().await` — single-PID SIGKILL).
- **Invariants violated:** #1 (timeout must terminate the child), #10 (`kill_on_drop` must leave no orphaned child). Trust boundary #1.

**Failure scenario.** A strategy `command: "yadm push"` runs `sh -c`, which forks/execs `yadm`, which spawns `ssh` that hangs on an unresponsive network. When `resolved_timeout` expires, `child.kill()` sends SIGKILL to the `sh` (or `yadm`) process only. tokio's `kill` and `kill_on_drop` both act on the single child PID, never the process group, so the grandchild `ssh` is reparented to init and keeps running indefinitely. This is precisely the 9.5h yadm-SSH hang the project exists to prevent: the timeout fires, crosshair reports `timed_out=true`, yet the actually-hung process is never terminated. Repeated ticks accumulate orphaned hung processes.

**Evidence.** `src/runner.rs:67-73` — spawn with no `process_group`, `kill_on_drop(true)` only; `:110` and `:124` — `child.kill().await` = single-PID SIGKILL. Repo-wide grep finds zero `process_group`/`setpgid`/`killpg`, and `Cargo.toml` has no `nix`/`libc`. Reachable via the production loop `tick.rs:63 → 84 → 98` (`run_one_tick → converge_one → run_attempt`). `README.md:33` names yadm dotfile sync as the canonical convergence target, matching the scenario. Both lenses CONFIRMED. (Invariant-validity lens noted a labelling nuance — invariant #1 read literally is satisfied for the *direct* child, and `kill_on_drop` never promised grandchild reaping — but the substantive defect, a hung descendant surviving the safety timeout, is real.)

> This finding is the "orphan survival" facet of the same root cause as F1. F1 additionally captures the loop-wedge; F2 stands alone because orphan accumulation is a real, separately-observable harm (resource leak, defeated safety guarantee) even in a hypothetical build where the drain were already bounded.

**Fix sketch.** Spawn each attempt in its own process group (`cmd.process_group(0)`, i.e. `setpgid` on the child). On timeout/drop, kill the whole group with `killpg(pgid, SIGKILL)` instead of `child.kill()`. Keep `kill_on_drop` as a backstop but rely on group kill for the tree. (Same change as F1(a)/F1(b).)

**Verification step.** Strategy `command: "sh -c 'sleep 600 & wait'"` with a 1s `strategy.timeout`. After `run_attempt` returns, assert no process in the attempt's pgid survives (`kill(-pgid, 0)` returns ESRCH). Regression guard: fail the test if any child of the daemon PID remains after the attempt.

### F3 — Unbounded pipe drain: a backgrounding or descendant-holding command stalls the tick loop forever

- **Severity:** High
- **Verdict:** CONFIRMED (reachability: confirmed · invariant-validity: confirmed)
- **Location:** `src/runner.rs:96-99` (timeout wraps only `child.wait()`), `:124-126` (kill then unbounded drain), `:142-155` (`read_to_string_capped` exits only on EOF or 64 KiB cap).
- **Invariants violated:** #1 (no strategy may run unbounded), #8 (a slow tick must not break correctness). Trust boundary #1.

**Failure scenario.** `timeout(limit, wait)` wraps only `child.wait()`. The subsequent `stdout_task.await` / `stderr_task.await` run with no time bound, and `read_to_string_capped` blocks until EOF or 64 KiB. If the command backgrounds a process or spawns a descendant that inherits the stdout/stderr fds and keeps them open (`command: "./deploy.sh &"`, `"sleep infinity &"`, or any daemonizing/ssh-forking strategy producing <64 KiB), `sh` exits immediately, `wait()` returns `Ok` **inside** the timeout, but the drain await never observes EOF. `run_attempt` never returns, `converge_one` never returns, and the sequential tick loop is wedged permanently — every other target starves. The per-attempt timeout provides zero protection because it does not cover the drain. Even the timeout-branch kill only reaps the direct child, so a surviving grandchild still holds the pipe open and the following drain await hangs.

**Evidence.** `runner.rs:96` — `timeout(limit, wait)` wraps only `child.wait()`. `runner.rs:98-99, 111, 125-126` — unbounded drain awaits. `runner.rs:142-155` — read loop exits only on EOF or the 64 KiB cap. `runner.rs:89-93` — read-ends moved into spawned tasks while write-ends are inherited by any descendant. `runner.rs:124` — kill reaps only the direct child. `tick.rs:78-82` — sequential loop, no watchdog; `tick.rs:98` — `run_attempt` awaited with no outer timeout. Trigger is an ordinary user-authored backgrounding idiom in an `sh -c` strategy — realistic without any adversary. No refuting mechanism (overall timeout, process-group kill, or read deadline) exists in the code. Both lenses CONFIRMED.

> F3 is the "background-stall" facet of the same root cause: it triggers even on the **happy path** (`Ok(Ok(status))`), where the timeout never fires at all, so it is distinct from F1 (which is the timeout/kill path). A complete fix must bound the drain regardless of which arm ran.

**Fix sketch.** Bound the drain: wrap the `stdout_task`/`stderr_task` joins in their own `timeout(...)`, aborting the read tasks on expiry — on **all three** arms (`Ok(Ok)`, `Ok(Err)`, `Err`). Combined with process-group kill (F1/F2), killing the whole group closes all pipe write-ends and yields prompt EOF. Never `.await` a child-fed pipe read outside a deadline.

**Verification step.** Strategy `command: "sleep infinity &"` with a default timeout; assert `run_attempt` returns within a small bound (not blocked on the backgrounded child), that the outcome is recorded, and that a second target in the same tick advances. Complements the F1 test, which drives the timeout arm; this one drives the happy arm.

### F4 — SQLite opened with no `busy_timeout` or WAL; contention with `status` drops the persist and reintroduces busy-loop hammering

- **Severity:** High
- **Verdict:** CONFIRMED (reachability: confirmed · invariant-validity: confirmed)
- **Location:** `src/store.rs:60` — `Store::open` (`:55-65`) and `migrate` (`:74-92`) issue no `PRAGMA busy_timeout` or `PRAGMA journal_mode=WAL`.
- **Invariants violated:** #5 (backoff prevents busy-loop hammering). Trust boundary #8.

**Failure scenario.** `Store::open` and `migrate` never set `busy_timeout` or WAL, so rusqlite defaults apply: `busy_timeout=0` (immediate `SQLITE_BUSY`) and rollback-journal mode (a writer needing RESERVED→EXCLUSIVE cannot upgrade while any reader holds SHARED). When a user runs `crosshair status` — which opens the same DB and iterates `get_or_empty` per target, each holding a brief SHARED lock — at the instant a tick calls `record_attempt`, the write returns `SQLITE_BUSY` immediately with no retry. That error propagates through `record_attempt(...)?` at `tick.rs:112`, so `converge_one` returns `Err` and `run_one_tick` merely logs a warning (`tick.rs:79-81`) — **the outcome is never persisted**: `last_attempt_at`, `consecutive_failures`, and `cooldown_until` all stay at their prior values. On the next tick, `get_or_empty` returns the unchanged row (cooldown still None), so the failing strategy is re-run with **no backoff at all** — exactly the hammering invariant #5 exists to prevent — for as long as a `status` process periodically contends. Symmetrically, `status` itself can abort with "database is locked" while the daemon holds the write lock.

**Evidence.** `store.rs:55-65` (open, no pragmas) and `:74-93` (migrate, no pragmas); repo-wide grep shows zero `busy_timeout`/`journal_mode`/`PRAGMA`. Shared DB: `tick.rs:40-41` and `status.rs:16-17` both call `resolved_state_path()` + `Store::open`. Reader locks: `status.rs:30-31` (`get_or_empty` per target). Writer BUSY: `store.rs:140` single autocommit INSERT with default `busy_timeout=0`. Drop path: `record_attempt?` at `tick.rs:112 → converge_one Err → warn-only at tick.rs:79-81 →` cooldown never persisted `→ tick.rs:90` re-runs next tick. Both lenses CONFIRMED.

> **Confidence caveat (from both verdicts).** The collision window is sub-millisecond, so the finding's strongest phrasing — hammering "every single tick" for a sustained period — overstates the magnitude: `status` is a brief, human-invoked CLI, so realistically this yields *occasional* dropped persists and early re-runs during narrow contention windows, not continuous hammering. Sustained hammering would require contention on every tick. This concerns magnitude, not existence; the invariant violation and the reachable data-loss path are real. Severity is retained at High because a single dropped first-failure persist silently defeats backoff for that target until the next successful write, and the daemon is explicitly designed to run alongside ad-hoc `status` inspection.

**Fix sketch.** In `Store::open`, immediately after `Connection::open`, set `conn.busy_timeout(Duration::from_secs(5))` and `conn.pragma_update(None, "journal_mode", "WAL")` so readers and the writer no longer block each other and transient contention retries rather than erroring. Additionally, treat a persist failure in `converge_one` as a reason to still apply an in-memory/next-tick cooldown, so a lost write can never produce a backoff-free re-run.

**Verification step.** Test: open two connections to one temp DB; hold a SHARED read transaction on one; attempt `record_attempt` on the other. Without the fix, assert it errors `SQLITE_BUSY`; with the fix (busy_timeout + WAL), assert it succeeds. Integration guard: spawn a tight `status`-poll loop against a running one-shot daemon over a failing strategy and assert `consecutive_failures` and `cooldown_until` advance monotonically (no reset-to-None between ticks).

## Proposed bullseye targets

Umbrella: **T5.x crosshair Fable audit**. These are proposals only — not filed, not committed.

- **T5.x.1** — All confirmed process-management defects (F1/F2/F3) are closed: strategies spawn in their own process group, timeout/drop kills the whole group, and pipe drains are bounded on every arm. Acceptance: integration tests prove `run_attempt` returns within `timeout + epsilon` and leaves no surviving descendant for (a) a hung command with a pipe-holding grandchild, (b) a backgrounding command on the happy path, and (c) a subsequent target in the same tick still executes.
- **T5.x.2** — The store no longer drops attempt persists under cross-process contention: `Store::open` sets WAL + a non-zero `busy_timeout`, and a persist failure still yields a next-tick cooldown. Acceptance: concurrent `status`-poll + failing-strategy test shows `consecutive_failures`/`cooldown_until` advance monotonically with no backoff-free re-runs.
